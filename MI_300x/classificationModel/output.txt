services:
  classification_model :
    container_name: classification_model
    image: nayanmoni458/stoneinscription:rocm-test 
    ports:
      - "8001:8001"
    devices:
      - /dev/kfd:/dev/kfd
      - /dev/dri:/dev/dri
    group_add:
      - video
    security_opt:
      - seccomp:unconfined
    cap_add:
      - SYS_PTRACE
    ipc: host
    shm_size: 16gb
    ulimits:
      memlock: -1
      stack: 67108864
    env_file:
      - .env
    volumes:
      - ./models:/app/models
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8001/"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
    networks:
      - monitoring-network

  rocm_exporter:
    image: rocm/device-metrics-exporter:nic-v1.0.0
    container_name: rocm_exporter
    restart: unless-stopped
    ports:
      - "9835:9101"
    devices:
      - /dev/kfd
      - /dev/dri
    group_add:
      - video
    networks:
      - monitoring-network

networks:
  monitoring-network:
    driver: bridge

===== ./README.md =====
# Classification Model — Stone Inscription

ROCm-based classification inference service for the Stone Inscription platform, exposing a REST API on port `8001` with GPU passthrough and Prometheus-compatible GPU metrics via a ROCm device exporter on port `9835`.

---

## Prerequisites

- AMD GPU server running Linux with ROCm drivers installed
- Docker Engine with Compose v2 plugin (`docker compose` not `docker-compose`)
- `/dev/kfd` and `/dev/dri` available on the host (confirms ROCm kernel modules are loaded)
- Your user added to the `video` group on the server:
  ```bash
  sudo usermod -aG video $USER
  newgrp video
  ```
- `scp` or SSH access to the target server from your local machine

---

## Target Directory Structure on the VM

All files must live under the following path on the server:

```
/home/<your-user>/artifactRegistory/Classification-Model/
├── docker-compose.yml
├── .env
└── models/               # Model weights — must exist before starting the stack
```

> **Note:** The parent directory name is `artifactRegistory` (with that exact spelling/casing). Use it exactly as shown.

---

## Step 1 — Create the Directory on the VM

SSH into the server and create the full directory structure:

```bash
ssh <user>@<server-ip>

mkdir -p /home/<your-user>/artifactRegistory/Classification-Model/models
```

The `models/` subdirectory must exist before starting the stack — Docker will mount it into the container at `/app/models`.

---

## Step 2 — Transfer Files from Your Local Machine

From your **local machine**, copy `compose.yml` and `.env` into the target directory on the VM:

```bash
# Copy the compose file (rename it to docker-compose.yml on the server)
scp compose.yml <user>@<server-ip>:/home/<your-user>/artifactRegistory/Classification-Model/docker-compose.yml

# Copy the environment file
scp .env <user>@<server-ip>:/home/<your-user>/artifactRegistory/Classification-Model/.env
```

If your model weights are stored locally and need to be transferred as well:

```bash
# Copy a single model file
scp /path/to/your/model.bin <user>@<server-ip>:/home/<your-user>/artifactRegistory/Classification-Model/models/

# Or copy an entire local models/ directory recursively
scp -r ./models/ <user>@<server-ip>:/home/<your-user>/artifactRegistory/Classification-Model/models/
```

---

## Step 3 — Environment Variables

The `.env` file you transferred contains the following variables, which are injected directly into the `classification_model` container at startup:

```env
HSA_OVERRIDE_GFX_VERSION=9.4.2
ROCM_PATH=/opt/rocm
HIP_VISIBLE_DEVICES=0
CUDA_VISIBLE_DEVICES=0
HSA_ENABLE_SDMA=0
```

| Variable | Value | Purpose |
|---|---|---|
| `HSA_OVERRIDE_GFX_VERSION` | `9.4.2` | Overrides the GFX version reported to ROCm — required when the installed ROCm version does not natively recognise the GPU |
| `ROCM_PATH` | `/opt/rocm` | Tells the runtime where ROCm is installed on the host |
| `HIP_VISIBLE_DEVICES` | `0` | Restricts HIP to GPU index 0 |
| `CUDA_VISIBLE_DEVICES` | `0` | Restricts CUDA compatibility layer to GPU index 0 |
| `HSA_ENABLE_SDMA` | `0` | Disables SDMA (System DMA) — improves stability on some AMD GPU generations |

Do not remove or rename this file. The container will start but the GPU runtime will likely fail without these variables.

---

## Step 4 — Create the Docker Network

Both services share a bridge network called `monitoring-network`. Create it once on the server before the first run:

```bash
docker network create monitoring-network
```

If the network already exists from a previous setup, this command will return an error which you can safely ignore. If you skip this step entirely, `docker compose up` will fail with a network-not-found error.

---

## Step 5 — Start the Stack

SSH into the server, navigate to the project directory, and bring the stack up:

```bash
ssh <user>@<server-ip>

cd /home/<your-user>/artifactRegistory/Classification-Model

docker compose up -d
```

This starts two containers:

| Container | Image | Host Port | Purpose |
|---|---|---|---|
| `classification_model` | `nayanmoni458/stoneinscription:rocm-test` | `8001` | Inference API |
| `rocm_exporter` | `rocm/device-metrics-exporter:nic-v1.0.0` | `9835` | GPU metrics (Prometheus) |

---

## Verifying the Services

### Built-in health check

Docker polls `http://localhost:8001/` every 30 seconds (10 s timeout, 3 retries, 40 s grace period on startup). Check the current status:

```bash
docker inspect --format='{{.State.Health.Status}}' classification_model
```

Expected output: `healthy`

### Manual endpoint check

```bash
curl -f http://localhost:8001/
```

### ROCm metrics endpoint

```bash
curl http://localhost:9835/metrics
```

### Logs

```bash
# Follow live logs
docker compose logs -f classification_model

# Last 100 lines
docker compose logs --tail=100 classification_model
```

## Stopping and Cleanup

```bash
# Stop containers, preserve volumes and network
docker compose down

# Stop containers and remove volumes
docker compose down -v

# Remove orphaned containers from previous runs
docker compose down --remove-orphans
```

---

## GPU and System Configuration Reference

The `classification_model` container runs with elevated privileges required by the ROCm runtime. These are all intentional — do not strip them:

| Setting | Value | Reason |
|---|---|---|
| Devices | `/dev/kfd`, `/dev/dri` | AMD GPU kernel driver access |
| Group | `video` | Required for DRI device access |
| `seccomp` | `unconfined` | ROCm runtime requirement |
| `cap_add` | `SYS_PTRACE` | ROCm debugging and profiling support |
| `ipc` | `host` | Shared memory with host processes |
| `shm_size` | `16gb` | Large model inference buffers |
| `ulimit memlock` | `-1` (unlimited) | GPU memory locking |
| `ulimit stack` | `64 MB` | Extended stack for native ROCm libs |

---

## Restart Behaviour

Both containers are set to `restart: unless-stopped`. They will automatically come back up after host reboots or Docker daemon restarts, unless explicitly stopped with `docker compose down`.

---

## Troubleshooting

**Container exits immediately**

```bash
docker compose logs classification_model
```

Common causes: missing model files in `models/`, incorrect `.env` values, or GPU not accessible.

**`monitoring-network` not found**

```bash
docker network create monitoring-network
```

Then retry `docker compose up -d`.

**`/dev/kfd: no such file or directory`**

The ROCm kernel module is not loaded on the host. Verify:

```bash
ls /dev/kfd
```

If missing, the ROCm driver stack needs to be installed or the host needs a reboot after installation.

**`HSA_OVERRIDE_GFX_VERSION` errors in logs**

The value `9.4.2` is set for the AMD MI300x. If you are running a different GPU generation, this value needs to match your GPU's GFX version. Check your GPU's GFX version with:

```bash
rocminfo | grep gfx
```

**Port already in use**

```bash
ss -tlnp | grep -E '8001|9835'
```

Stop whichever process is holding the port before starting the stack.

**Permission denied on `/dev/dri` or `/dev/kfd`**

Confirm your user is in the `video` group and that you started a new shell session after adding them:

```bash
groups | grep video
```
